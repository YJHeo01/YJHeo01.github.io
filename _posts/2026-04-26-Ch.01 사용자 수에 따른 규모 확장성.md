# Chapter 1. 사용자 수에 따른 규모 확장성

이번 챕터에서는 1명의 사용자를 지원하는 시스템부터, 최종적으로는 몇백만 사용자를 지원하는 시스템을 다룬다고 한다.

## 단일 서버 

단일 서버란 모든 컴포넌트가 단 한 대에서 실행되는 간단한 시스템

이 시스템에서의 사용자 요청이 처리되는 흐름은 다음과 같다.

1. 클라이언트가 도메인 이름을 통해 DNS 서버에서 ip 주소를 받아온다.
2. 클라이언트가 해당 IP 주소로 HTTP 요청을 전달한다.
3. 요청을 받은 웹 서버는 HTML이나 JSON 형태의 응답을 반환한다.

요청은 두 가지 단말에서 온다고 한다.

1. 웹 어플리케이션
2. 모바일 앱

## 데이터베이스

서버 하나로 사용자를 감당할 수 없을 경우, 여러 서비를 두어야 하는데, 웹 서버와 데이터베이스 서버로 분리한다고 한다.

### 관계형 데이터베이스 vs 비-관계형 데이터베이스

관계형 데이터베이스 (RDBMS)
- MySQL, 오라클 데이터베이스, PostgreSQL
- 자료를 테이블과 열, 칼럼으로 표현
- join 연산 가능

비 관계형 데이터베이스 (NoSQL)
- 키-값 저장소, 그래프 저장소, 칼럼 저장소, 문서 저장소로 분류
- 일반적으로 join 연산 지원하지 않음

대부분의 경우, 관계형 데이터베이스를 사용, 단 아래와 같은 경우 비-관계형을 고려할 수 있다.
- 아주 낮은 응답 지연시간이 요구됨
- 다루는 데이터가 비정형이라 관계형 데이터가 아님
- 데이터를 직렬화 혹은 역직렬화만 하면 되는 경우
- 아주 많은 양의 데이터를 저장할 필요가 있음

## 수직적 규모 확장 vs 수평적 규모 확장
 
수직적 규모 확장의 특징
- 서버에 고사양 자원을 추가하는 행위이다.
- 단순해서 트래픽의 양이 적을 때는 좋은 선택이다.
- 한 대의 서버에 자원을 무한대로 증설할 수 없다는 한계가 존재
- 자동복구 및 다중화 방안이 존재하지 않음

=> 대규모 트래픽을 처리할 때는 수평적 규모 확장이 적절하다.

### 로드밸런서
- 트래픽 부하를 분산한다.
- 클라이언트가 로드밸런서의 공개 IP 주소로 접속함 -> 따라서, 웹 서버는 클라이언트 접속을 직접 처리하지 않음
- 로드밸런서는 웹 서버와 통신하기 위해 사설 IP 사용

사설 ip란? 같은 네트워크에 속한 서버 사이의 통신에서 사용하는 ip 주소
- 10.0.0.0 ~ 10.255.255.255
- 172.16.0.0 ~ 172.31.255.255
- 192.168.0.0 ~ 192.168.255.255

### 데이터베이스 다중화

- 서버 간 master - slave 관계를 설정하고, 원본을 주 서버에, 사본을 부 서버에 저장하는 방식
- 쓰기 연산은 master에서만 지원, 읽기 연산은 slave에서만 지원
- 가용성(일부 장애시에도 정상 작동) 및 안정성(일부 서버 훼손되도 데이터는 보존될 것) 증가
- 단 하나 뿐인 slave가 다운 시, 읽기 연산도 모두 master가 처리할 것
- master가 다운될 경우, slave가 master 역할을 수행

번외 : 현직자와 대화를 하다가 Active–Standby라는 구조도 있다는 것을 알게 되었다.

## 캐시

애플리케이션의 성능은 데이터베이스를 얼마나 자주 호출하느냐에 좌우된다.
따라서, 값비싼 연산 결과를 요하거나 자주 참조되는 데이터를 메모리에 두고, 요청을 보다 빠르게 처리하는 저장소인 캐시를 사용한다.

### 캐시 계층

데이터가 잠시 보관되는 계층, 다음 설명은 캐시 서버의 작동 방식 중 하나입니다.

1. 데이터가 캐시에 있는 경우 : 캐시에서 데이터를 반환
2. 데이터가 캐시에 없는 경우 : DB에서 캐시에 데이터를 씀, 이후 반환

이러한 캐시 전략을 읽기 주도형 캐시 전략이라고 합니다. 그 외에도 다른 캐시 전략이 존재합니다.


### 캐시 사용 시 유의 사항

- 데이터 갱신이 잦지 않지만, 참조가 빈번하게 일어날 경우 고려할만 하다.
- 캐시에 적합한 데이터 : 캐시는 휘발성 메모리를 사용하므로, 영속적 보관이 필요한 데이터는 적합치 않음
- 만료 정책에 대해서도 고려해야, 너무 짧으면 데이터베이스 접근 증가, 너무 길면 원본과 차이가 발생
- 일관성에 대해서도 고려해야 함 (저장소의 원본과 캐시 내의 사본의 일치) -> 캐시 갱신과 원본 갱신이 단일 트랜잭션으로 처리되지 않을 경우 깨질 수 있음
- 장애 대처 방안, 캐시 서버가 단 하나일 경우 SPOF가 될 가능성이 존재 -> 이를 회피하기 위해 캐시 서버 분산 필요
- 캐시 메모리의 크기 -> 너무 작으면 캐시 성능이 떨어짐 -> 과할당하면 갑자기 늘어도 커버 가능
- 데이터 방출 정책 : 캐시가 꽉 찰 경우 내보낼 데이터를 정하는 규칙 (LRU, LFU, FIFO 등)

### 읽고 든 생각

이 파트를 읽으며 BOJ_STAT 프로젝트에서 캐시에 관해서 이것저것 고민했던 것이 떠올랐다. 엄밀히 말하면, 내 BOJ_STAT에서는 캐시 서버를 사용한 것이 아니고, CDN 캐시 헤더를 설정해서 TTL을 조절해주었던 것이지만.... 이에 대해서도 한번 더 정리하는 시간을 가져봐야 겠다. 백준 온라인 저지가 서비스 종료를 앞두며 이 프로젝트 또한 중단할 예정이었지만, 학습한 내용을 바탕으로 해당 프로젝트의 아키텍처를 개선하는 방법에 대해 한번 고민해볼 수 있을듯 싶다.

또한, 대부분의 경우에서는 가장 마지막에 사용한 콘텐츠를 제공하는 LRU와 같은 방식이 효율적이라고 생각하는데, LFU, FIFO 같은 방식은 어떤 경우에 사용하는지도 궁금했고, 이에 대해 한번 고민해보았다. 내 추론에는 LFU의 경우에는 캐시에 할당된 프로그램이 변경되는 경우가 매우 드물 때, FIFO는 이와 반대인 상황에서 유용하지 않을까 싶다라는 생각이 들었다.

그래서 조금 더 찾아보니 다음과 같은 내용을 확인할 수 있었다.

LFU는 상위 1% 데이터가 트래픽의 대부분을 차지하고, 인기 콘텐츠가 자주 바뀌지 않은 경우에 유용하다고 한다. 하지만 과거에 인기있었지만, 현재에는 인기없는 컨텐츠를 거르기 위해 시간 가중치도 같이 부여하여 사용한다고 한다.

FIFO의 경우에는 가장 단순한 전략이기에 정확한 캐싱보다 처리량이 중요한 시스템에서 주로 사용한다고 한다.

## 콘텐츠 전송 네트워크(CDN)

지리적으로 분산된 서버의 네트워크, 이미지, 비디오, CSS, JS 등 정적 콘텐츠를 캐시한다.

CDN의 동작 예시는 다음과 같다.

1. 클라이언트 A가 image.jpg 액세스
2. CDN에 이미지가 존재하지 않아 원본 서버에서 가져 옴
3. 이미지를 지정된 TTL만큼 CDN에 저장 및 A에게 반환
4. TTL에 명시된 시간이 끝나기 전 클라이언트 B가 image.jpg 요청, CDN에 캐시된 이미지 반환

### CDN 사용 시 고려 사항

- 비용 : 제3 사업자에 의해 운영되며, 전송량에 따라 요금을 내야 함, 따라서 자주 사용하지 않은 콘텐츠는 빼는 것을 고려
- 적절한 만료 시한 설정 : 시의성에 영향을 줌, 너무 짧으면 원본 접속 증가, 너무 길면 콘텐츠 신선도 하락
- CDN 장애에 대한 대처 방안
- 콘텐츠 무효화 방법 (CDN 사업자가 제공하는 API 사용 or 오브젝트 버저닝(버전 번호 인자))

### 읽고 든 생각

우선, BOJ_STAT이 계속 생각났다.. CDN 캐시 만료 시한 설정에 대한 이슈를 겪었기에 ㅋㅋ

그리고, CDN 장애에 대한 대처 방안이라면 결국, 해당 문제를 감지할 때 원본 서버에 접근하는 것인데, 이를 어떻게 구성하는 건지 조금 궁금하다. 이에 대해 찾아보니 DNS레벨이서 CDN을 우회하는 Origin 경로나 Backup CDN 경로를 미리 만들어둔다고 한다.

## 무상태(stateless) 웹 계층

웹 계층을 수평적으로 확장하기 위해 고민해보자.
이를 위해서는 사용자 세션 데이터 등 상태 정보를 웹 계층에서 제거해야 한다.
상태 정보를 관계형 데이터베이스나 NoSQL 같은 지속성 저장소에 보관하고, 필요하도록 가져오도록 구성하는 웹 계층을 무상태 웹 계층이라고 한다.

### 상태 정보 의존적인 이카텍처

상태 정보를 보관하는 서버는 상태를 유지하여 요청들 사이에 공유하도록 한다.

다음과 같은 아키텍처를 보자.

- 사용자 A - 서버 1 (사용자 A에 대한 세션 데이터, 사용자 A의 프로필 이미지)
- 사용자 B - 서버 2 (사용자 B에 대한 세션 데이터, 사용자 B의 프로필 이미지)
- 사용자 C - 서버 3 (사용자 C에 대한 세션 데이터, 사용자 C의 프로필 이미지)

같은 클라이언트으로부터의 요청은 항상 같은 서버로 전송되어야 한다. 위 상황에서 사용자 A의 요청이 서버 2로 전송되면 인증이 실패된다. 이를 지원하기 위해 로드밸런서에서 고정 세션을 제공하는데, 이는 로드밸런서에 부담을 주며 로드밸런스 뒷단에 서버를 추가하거나 제거하기도 까다로워 진다.

### 무상태 아키텍처

이 아키텍처의 구조는 다음과 같다.

사용자 A, B, C -> 웹 서버 -> 공유 저장소

- 사용자들은 웹 서버에 http 요청
- 웹 서버는 공유 저장소에 접근

## 데이터 센터

다중 데이터센터 아키텍처에서 해결해야 하는 기술적 난제들

- 트래픽 우회 : 올바른 데이터센터로 트래픽을 보내는 효과적인 방법을 찾아야 한다.
- 데이터 동기화 : 장애 발생 시 다른 데이터 센터로 우회했을 때, 찾는 데이터가 없을 수도 있음
- 테스트와 배포 : 위치마다 접근하는 데이터센터가 다르니 여러 위치에서 테스트를 해봐야 한다. 그리고, 자동화된 배도 도구로 모든 센터에 동일한 서비스가 설치되도록 해야한다.

## 메시지 큐

- Producer와 Consumer를 분리하여 비동기 통신을 지원하는 컴포넌트다.
- 메시지의 버퍼 역할을 하며, 비동기적으로 전송한다.
- 이를 이용하면 서비스 혹은 서버 간 결합이 느슨해져서 확장을 용이하게 한다
- 요청 흐름을 블로킹하지 않도록 작업을 분리

---

## 로그, 메트릭 그리고 자동화

- 로그 : 서버 단위 모니터링도 되지만, 단일 서비스로 모아주는 도구를 활용하면 조금 더 편리
- 자동화 : 시스템이 커질수록 생산성을 높이기 위해 자동화 도구 사용, 지속적 통합 도구 활용시 검증 절차를 자동으로 거치게 하여 문제를 쉽게 감지 가능 그 외에도 빌드, 테스트, 배포 등의 절차 또한 자동화 가능
- 메트릭 : 시스템의 상태를 관찰하는 것

### 유용한 메트릭

- 호스트 단위 메트릭 : CPU, 메모리, 디스크 I/O 등
- 종합 메트릭 : 데이터베이스 계층의 성능, 캐시 계층의 성능 등이 이에 해당
- 핵심 비즈니스 매트릭 : 일별 능동 사용자(DAU), 수익, 재방문 등

읽으면서 결국에는 이런 메트릭을 종합하여 추론하는 사고력이 필요하지 않나라는 생각이 든다.

## 데이터베이스의 규모 확장

### 수직적 확장

기존 서버에 고성능의 자원을 증성하는 것, 다만 다음과 같은 한계가 존재
- 서버 하드웨어에는 한계가 있으니 무한한 증설 불가능
- SPOF 위험성
- 비용

### 수평적 확장

다른 말로는 샤딩이라고도 부른다.

샤딩은 대규모 데이터베이스를 샤드라고 부르는 작은 단위로 분할하는 기술이다. 사용자 데이터를 어느 샤드에 넣을지는 사용자 ID를 토대로 정한다. 예를 들어, user_id % 4를 해시 함수로 사용한다던가 하는 방식이다. 샤딩 키를 정할 때는 데이터가 고르게 분할될 수 있도록 정하는 것이 중요하다.

샤딩을 도입할 경우 다음과 같은 문제에 직면할 수도 있다

- 유명인사 문제 : 특정 샤드에만 질의가 집중되어 서버에 과부하가 발생하는 문제다.
- 데이터가 너무 많아서 하나의 샤드만으로는 부족하거나, 샤드간 데이터 분포가 균등하지 못한 경우 샤드 키를 계산하는 함수를 변경하고 재배치를 해야 한다. 학습자료 뒷부분에서 다루는 안정 해시를 활용하면 이를 해결할 수 있다고 한다.
- 조인과 비정규화 : 하나의 데이터베이스를 여러 샤드 서버로 쪼개면, 여러 샤드에 걸친 데이터를 조인하기가 힘들어진다. 이를 해결하는 한 가지 방법은 데이터베이스를 비정규화하여 하나의 테이블에서 질의가 수행될 수 있도록 하는 것이다.

### 지금까지 살펴본 기법

- 웹 계층은 무상태 계층으로
- 모든 계층에 다중화 도입
- 가능한 한 많은 데이터를 캐시할 것
- 여러 데이터 센터를 지원할 것
- 정적 콘텐츠는 CDN을 통해 지원할 것
- 데이터 계층은 샤딩을 통해 그 규모를 확장할 것
- 각 계층은 독립적 서비스로 분할할 것
- 서비스를 지속적으로 모니터링하고, 자동화 도구들을 활용할 것
