# DNS/SNI 차단과 HTTP 451: 인터넷 접속 차단 방식 비교

최근 국내에서 Cloudflare의 HTTP 451 응답을 통한 차단 대상 사이트 접속 제한 사례가 알려지면서, CDN 레벨에서 웹사이트를 차단하는 것이 주목받았습니다. 따라서, 기존 접속 차단 방식과 비교했을 때 차단을 구현하는 원리가 궁금해져서 이를 탐구해보았습니다.

이번 글에서는 DNS 차단, HTTP 리다이렉트 방식, SNI 차단과 HTTP 451 기반 CDN 차단이 어떻게 다른지 정리해보고자 합니다.

> 이 글은 특정 사이트의 우회 방법을 설명하기 위한 글이 아니라, 인터넷 접속 차단이 네트워크 구조상 어느 지점에서 이루어지는지 이해하기 위한 기술 정리 글입니다.

---

## 1. 기존 인터넷 차단 방식

기존의 인터넷 접속 차단은 대체로 사용자의 요청이 목적지 서버에 도달하기 전에, 국내 ISP 또는 네트워크 구간에서 차단하는 방식으로 이해할 수 있으며, 대표적으로 다음과 같은 방식이 존재합니다.

- HTTP 리다이렉트
- DNS 차단
- SNI 기반 HTTPS 차단

### 1.1 HTTP 리다이렉트

HTTP의 주요 헤더 필드 중 Host가 존재합니다. Host 헤더에는 요청을 보낼 도메인 네임 정보가 있습니다.

또한, HTTP는 암호화되지 않은 프로토콜입니다. 따라서 사용자가 어떤 사이트에 접속하려는지 비교적 쉽게 확인할 수 있습니다.

따라서, Host 헤더를 확인한 이후, 차단 대상 사이트임이 확인되면 차단 안내 페이지로 연결해주는 것입니다.

그러나, HTTPS가 보편화되면서 이 방법의 한계가 발생했습니다. HTTPS에서는 HTTP 요청 내용이 TLS로 암호화되기 때문에, 중간 장비가 Host 헤더나 요청 경로를 그대로 볼 수 없기 때문입니다.


### 1.2 DNS 차단

사용자가 브라우저에 도메인 주소를 입력하면, 해당 도메인에 대응되는 IP 주소를 알아내야 합니다. 이때 사용되는 것이 DNS입니다.

DNS의 일반적인 흐름은 다음과 같습니다.

```text
사용자 → DNS 서버: example.com의 IP 주소를 알려줘
DNS 서버 → 사용자: A.B.C.D
사용자 → A.B.C.D로 접속
```

사용자가 요청한 주소가 차단 대상인 경우 다음과 같이 동작합니다.

```text
사용자 → DNS 서버: blocked-site.example의 IP 주소를 알려줘
DNS 서버 → 사용자: 차단 안내 페이지 IP 주소를 반환
사용자 : warning.or.kr 등 차단 안내 페이지 접속
```
즉, DNS 차단은 도메인 주소를 IP로 변환할 때, DNS 서버에서 차단 안내 페이지로 연결되는 주소를 반환하는 방식입니다.

이 방식의 경우, 사용자가 통신사 기본 DNS가 아닌 외부 DNS를 이용하면 회피할 수 있다는 한계가 존재했습니다.

### 1.3 SNI 기반 HTTPS 차단

HTTPS에서는 요청 내용이 암호화됩니다. 하지만 TLS 연결 초기 단계에서는 서버가 적절한 인증서를 선택할 수 있도록, 클라이언트가 접속하려는 도메인 정보를 알려주는 SNI(Server Name Indication)가 사용됩니다.

흐름을 단순화하면 다음과 같습니다.

```text
사용자 → 서버: TLS 연결을 시작하겠습니다.
사용자 → 서버: SNI = blocked-site.example
서버 → 사용자: 해당 도메인에 맞는 인증서 제공
```

SNI 기반 차단은 이 TLS 초기 연결 과정에 포함된 SNI 값을 보고, 접속하려는 도메인이 차단 대상인지 판단하는 방식입니다.

```text
사용자 → ISP → TLS ClientHello 확인
ISP: SNI가 차단 대상 도메인인지 확인
ISP: 차단 대상이면 연결 차단
```

SNI 차단 역시 TLS 초기 패킷 처리 방식에 따라 중간 장비가 SNI 값을 안정적으로 파싱하지 못하는 경우 차단 효과가 약해질 수 있다는 한계가 있습니다. 이에 대한 자세한 내용은 글의 주제를 벗어나므로 다루지 않겠습니다.

## 2. CDN을 사용하는 사이트의 요청 흐름

Cloudflare와 같은 CDN을 사용하는 사이트는 사용자가 원본 서버에 직접 접속하지 않는 경우가 많으며, 일반적인 흐름은 다음과 같습니다.

```text
사용자 → Cloudflare 엣지 → 원본 서버
```
여기서 Cloudflare 엣지는 사용자와 원본 서버 사이에 위치한 리버스 프록시 역할을 합니다.

즉, 사용자는 사이트의 원본 서버에 직접 요청을 보내는 것이 아니라, 먼저 가까운 Cloudflare 엣지 서버에 요청을 보냅니다. Cloudflare는 해당 요청을 받아 캐시된 응답을 반환하거나, 필요한 경우 원본 서버에 요청을 전달합니다.

```text
사용자 요청
  ↓
Cloudflare 엣지
  ↓
캐시된 응답이 있으면 바로 응답
  ↓
없으면 원본 서버로 요청 전달
  ↓
원본 서버 응답을 사용자에게 전달
```

중요한 점은 CDN이 단순히 정적 파일을 캐싱하는 서버만이 아니고 다음과 같은 역할도 수행할 수 있다는 것입니다.

- 캐싱
- DDoS 방어
- WAF 적용
- TLS 종료
- 접근 제어
- 국가 또는 지역 기반 정책 적용
- 법적 요청에 따른 응답 제한

즉, CDN은 원본 서버 앞단에서 요청을 제어할 수 있는 정책 집행 지점이 될 수 있습니다.

## 3. HTTP 451 기반 차단

HTTP 451은 Unavailable For Legal Reasons를 의미합니다. RFC 7725에서는 HTTP 451을 “법적 요구의 결과로 서버가 리소스 접근을 거부하는 상태 코드”로 정의하고 있습니다. Cloudflare 문서 역시 451 상태 코드를 법적 조치나 제한으로 인해 요청한 리소스를 제공할 수 없을 때 사용하는 응답이라고 설명합니다.

따라서, CDN 기반 HTTP 451 차단은 다음과 같은 흐름으로 이해할 수 있습니다.

```text
사용자 → Cloudflare 엣지
Cloudflare 엣지: 이 요청은 법적 제한 대상인가?
Cloudflare 엣지 → 사용자: HTTP 451 응답
원본 서버: 요청을 받지 못함
```


## 4. CDN 기반 차단의 적용 방식

CDN 기반 차단은 구체적으로 어떤 기준으로 적용하느냐에 따라 다시 나눠볼 수 있겠습니다.

### 4.1 한국 Cloudflare 엣지에서만 차단하는 경우

첫 번째는 특정 국가의 Cloudflare 엣지에서만 차단하는 방식입니다.

예를 들어 한국 내 Cloudflare 엣지에서만 특정 도메인에 대해 HTTP 451을 반환한다고 가정해봅시다.

```text
한국 사용자 → 한국 Cloudflare 엣지 → HTTP 451
해외 사용자 → 해외 Cloudflare 엣지 → 정상 응답 가능
```
이 방식에서는 사용자가 어느 Cloudflare 엣지로 연결되는지가 중요해집니다. 대부분 CDN 엣지는 응답이 가장 빠르고 가까운 곳으로 연결됩니다. 한국 내 네트워크에서 접속하면 대체로 한국 또는 인근의 Cloudflare 엣지로 연결될 가능성이 높고, 해당 엣지에서 차단 응답이 반환될 수 있습니다.

### 4.2 접속자 IP 국가가 KR이면 전 세계 엣지에서 차단하는 경우

두 번째는 Cloudflare 엣지의 위치가 아니라, 접속자 IP의 국가를 기준으로 차단하는 방식입니다.

```
한국 IP 사용자 → 한국 Cloudflare 엣지 → HTTP 451
한국 IP 사용자 → 일본 Cloudflare 엣지 → HTTP 451
한국 IP 사용자 → 미국 Cloudflare 엣지 → HTTP 451
```

이 경우 핵심은 “어느 엣지에 접속했는가”가 아니라 “Cloudflare가 접속자의 IP를 어느 국가로 판단했는가”입니다.

이 방식은 한국 엣지에서만 차단하는 것보다 더 강력한 방식입니다. 사용자가 우연히 해외 Cloudflare 엣지로 연결되더라도, 접속자 IP가 한국으로 판정되면 차단될 수 있기 때문입니다.

저는 2가지 방안 중 "접속자 IP 국가가 KR이면 전 세계 엣지에서 차단"을 적용했을 가능성이 더 높다고 생각합니다.(물론 두 방식이 함께 적용되었을 가능성도 배제할 수는 없습니다.) CDN 기반 차단을 구현할 때 한국 Cloudflare 엣지에서만 차단하는 방식은 라우팅 경로에 따라 해외 이용자가 한국 엣지로 연결되는 경우에도 영향을 받을 수 있습니다. 반면 접속자 IP의 국가가 KR이면 전 세계 Cloudflare 엣지에서 차단하는 방식은 엣지 서버의 위치가 아니라 사용자의 출구 IP 국가를 기준으로 정책을 적용하므로, 국가 단위 접근 제한이라는 목적에는 더 자연스럽기 때문입니다.

물론, 외부에서 Cloudflare의 내부 정책을 알 수 없기에 단정짓기에는 어렵습니다.

## 5. 기술적 한계와 우회 가능성

CDN 기반 차단이 기존 ISP 차단과 다르다고 해서, 기술적으로 완벽한 차단이 되는 것은 아닙니다.

가장 대표적인 한계는 접속자 IP 판정입니다.

VPN을 사용하면 Cloudflare 입장에서 보이는 접속자 IP는 사용자의 실제 한국 IP가 아닌 VPN 서버의 IP입니다.

"한국 Cloudflare 엣지에서만 차단하는 경우, VPN을 사용해도 한국 Cloudflare 엣지에 연결된다면?" 이라는 의문을 가질 수도 있겠습니다. 

그러나, 해외 VPN 서버를 통해 접속하면 Cloudflare 입장에서는 해당 VPN 서버의 IP에서 요청이 들어온 것으로 보입니다. 이 경우 일반적으로는 VPN 서버와 네트워크상 가까운 Cloudflare 엣지로 연결될 가능성이 높고 따라서, 한국 Cloudflare 엣지에 연결될 가능성이 낮습니다.

(단, 실제 엣지 배정은 BGP 라우팅, 피어링, 네트워크 혼잡도 등에 따라 달라질 수 있습니다.)

## 6. 정리

| 방식 | 차단 위치 | 기준 정보 | 한계 |
|---|---|---|---|
| DNS 차단 | DNS 질의 단계 | 도메인 | 외부 DNS 사용 시 회피 가능성 |
| HTTP 차단 | HTTP 요청 단계 | Host, URL | HTTPS에서는 내용 확인 어려움 |
| SNI 차단 | TLS Handshake 단계 | SNI | ECH, SNI 처리 방식에 영향 |
| CDN 451 차단 | CDN 엣지 | 법적 요청, IP 국가, CDN 정책 | VPN/프록시, IP 판정 한계 |

기존의 인터넷 접속 차단과 HTTP 451 기반 CDN 차단은 결과적으로 사용자에게 “접속 불가”를 보여준다는 점에서는 비슷하지만 기술적으로 보면 차단 위치와 방식이 다릅니다.

기존 방식은 주로 국내 ISP 구간에서 이루어졌습니다.

```text
사용자 → 국내 ISP → 차단
```

반면 CDN 기반 HTTP 451 차단은 Cloudflare와 같은 CDN 엣지에서 요청을 처리하는 과정에서 발생합니다.

```text
사용자 → Cloudflare 엣지 → HTTP 451
```

즉, 기존 방식이 네트워크 사업자 중심의 차단이라면, CDN 기반 차단은 사이트 앞단에 위치한 글로벌 인프라 사업자가 정책 집행 지점이 되는 방식이라고 볼 수 있겠습니다.
